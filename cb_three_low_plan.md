# 可转债三低排序改造计划

## 目标

把当前 `cb_main.py` 的可转债排序逻辑，调整为与你提供的另一套服务端逻辑尽量一致的版本。

## 当前本地逻辑

当前 `cb_main.py` 的三低排序逻辑是：

- 过滤后，按 `price`、`curr_iss_amt`、`premium_rt` 三个字段分别排名
- 三项都按“越低越好”
- 使用“名次分直接相加”的方式计算总分
- 总分越低越靠前

特点：

- 第一因子用的是 `price`
- 处理了并列名次
- 当前只排除了部分强赎/赎回状态

## 目标逻辑

根据你提供的代码，目标逻辑应为：

- 先做筛选，再做打分
- 打分因子为：
  - `dblow`（双低）
  - `premium_rt`（转股溢价率）
  - `curr_iss_amt`（剩余规模）
- 三个因子都是负相关，即“越低越好”
- 每个因子的打分公式：

```text
score = (total - rank + 1) * weight
```

- 三个因子的分数相加得到总分
- 总分越高越靠前

## 关键差异

当前本地代码与目标逻辑的主要差异：

1. 第一因子不同
   - 当前：`price`
   - 目标：`dblow`

2. 汇总方式不同
   - 当前：名次直接相加，分数越低越前
   - 目标：名次换算为得分后相加，分数越高越前

3. 并列处理不同
   - 当前：相同值给相同名次
   - 目标代码：相同值也会按顺序拿不同 rank

4. 过滤条件不同
   - 当前本地过滤条件较少
   - 目标逻辑还包括：
     - 排除 ST
     - 排除新债
     - 排除评级
     - 排除触发进度
     - 排除强赎状态（已公告强赎、实施强赎、到期赎回、满足强赎）

## 建议改造范围

第一阶段先做“排序逻辑对齐”，暂不一次性搬全部复杂过滤，避免改动过大。

第一阶段建议：

1. 把排序因子从 `price` 改为 `dblow`
2. 把总分算法从“名次求和”改为“排名得分求和”
3. 把最终排序改为“总分降序”
4. 消息展示改为：
   - 双低得分
   - 溢价率得分
   - 规模得分
   - 总分
5. 保留现有 CSV 导出和预览脚本
6. 增加独立校验脚本，验证排序结果

第二阶段再考虑补齐过滤规则：

1. 排除 ST
2. 排除低评级/无评级
3. 排除实施强赎
4. 视需要补“满足强赎”判断

## 实施步骤

### 第一步：排序算法替换

在 `cb_main.py` 中：

- 新增按单因子生成“排名得分”的函数
- 改造总分计算逻辑
- 把 `price` 替换成 `dblow`
- 总排序改为按总分降序

### 第二步：消息展示调整

在消息中显示：

- 双低
- 转股溢价率
- 剩余规模
- 三低总分
- 各因子的排名/得分

### 第三步：验证脚本

新增或改造测试脚本，至少覆盖：

1. 原始数量、过滤后数量、排序后数量
2. 前 20 名预览
3. 某几只债的分数拆解
4. 与预期公式的独立校验

### 第四步：文档更新

同步更新：

- `README.md`
- `cb_ranking.md`

说明当前三低排序改成：

- 双低越低越好
- 溢价率越低越好
- 规模越低越好
- 按打分制汇总，而不是简单名次求和

## 风险点

1. 如果改成目标逻辑，当前排名结果会明显变化
2. 如果不补齐过滤条件，结果仍然不会与另一套服务端完全一致
3. “满足强赎”与“实施强赎”需要确认你是否要完全照搬
4. 目标代码的并列处理是顺序排名，不是并列同名次，需要确认是否保留这个行为

## 推荐落地方案

推荐先执行：

- 只改排序算法
- 保留当前过滤条件
- 先确认排序结果是否符合你的预期

确认没问题后，再继续补过滤规则。

这样更容易定位差异，也更不容易一次改太多。
